# ソフトマックス関数（Softmax）のメカニズムとグラフ特性

アテンションメカニズムでは、計算された生の類似度スコア（ドット積の結果など）を、**「合計がぴったり 1.0 (100%) になり、かつすべて 0 以上 1 以下の値になる確率分布」**に正規化する必要があります。

これを行うのが**ソフトマックス関数（Softmax）**です。本ドキュメントでは、ソフトマックスが描く数値のカーブ（グラフ特性）と、アテンションで使われる数学的な理由について解説します。

---

## 1. ソフトマックス関数の数式

リストの中の $i$ 番目の値 $x_i$ に対するソフトマックスの出力は次の式で計算されます。

$$\text{Softmax}(x_i) = \frac{e^{x_i}}{\sum_{j} e^{x_j}}$$

*   **分子 ($e^{x_i}$)**: ネイピア数 $e \approx 2.718$ を入力値の乗数とした指数関数です。
*   **分母 ($\sum e^{x_j}$)**: リスト内のすべての要素に対する分子の合計値です。これにより、全体の合計がぴったり 1.0 に正規化されます。

---

## 2. 指数関数 $y = e^x$ がもたらす「グラフの形状」

ソフトマックスの挙動を決定づけているのは、分子にある **指数関数 $y = e^x$** のグラフ特性です。

```text
    y (出力)
    │                                📈 急激に立ち上がる！
   8│                               *
   7│                              *
   6│                             *
   5│                            *
   4│                          *
   3│                        *
   2│                     *
   1│             *  *  *  (x=0のときy=1)
   0└───*──*──*───────────────── x (入力)
      -3  -2  -1  0  1  2  3
```

### 📈 指数グラフの特徴
1.  **常に正の値をとる**:
    入力 $x$ がどれほど大きなマイナスの値であっても、出力 $y$ は必ずゼロより大きい正の値（$y > 0$）になります（$x \to -\infty$ で $y \to 0$）。
2.  **右肩上がりに急激に爆発する**:
    $x$ が少し大きくなるだけで、値は掛け算的に爆発して大きくなります（例: $e^1 \approx 2.7$, $e^2 \approx 7.4$, $e^3 \approx 20.1$, $e^5 \approx 148.4$）。

---

## 3. なぜ単純な割合ではなく、ソフトマックスを使うのか？

単純に「合計値で割り算する」だけ（例: $x_i / \sum x_j$）ではなく、わざわざ指数関数 $e^x$ を噛ませるのには、LLMの学習と推論において極めて重要な3つの理由があります。

### ① 負の値を「正の確率」に変換できる
アテンションスコアはドット積で計算されるため、ベクトルの向きが反対であれば**マイナスの値（負のスコア）**になります。
もし単純な割り算をしてしまうと、以下のように確率として破綻します。
*   スコアが `[-2.0, 1.0, 3.0]` （合計 2.0）のとき：
    *   単純な割り算 $\to$ `[-1.0, 0.5, 1.5]` （確率がマイナスや1以上になり破綻）
    *   **ソフトマックス** $\to$ `[0.005, 0.113, 0.882]` （全て0〜1に収まり、合計も1.0になる）

### ② 最大値を際立たせる（Soft-Max の由来）
指数関数は「大きい値をより大きく、小さい値をより小さく」誇張します。これにより、スコアが僅かに高いだけの単語を、アテンションの対象として極端に際立たせることができます。
*   スコアが `[1.0, 2.0, 3.0]` （3.0が一番大きい）のとき：
    *   単純な割り算 $\to$ `[17%, 33%, 50%]` （3.0が半分を占める）
    *   **ソフトマックス** $\to$ `[9%, 24%, 67%]` （3.0が約7割を独占）

完全に最も大きいもの「だけ」を選ぶ関数を **ハードマックス (Hardmax)** と呼びますが、これだと一番以外の情報が完全に消えてしまいます。ソフトマックスは、他の可能性もわずかに残しつつ最大値を際立たせる（**ソフトな最大値**）ため、アテンションに最適な選択肢なのです。

### ③ 微分可能であり、学習（逆伝播）が安定する
ハードマックス（最大値だけを1にし、他を0にする）はグラフに「カド（折れ線）」ができるため、数学的に微分ができません。
ソフトマックスはグラフが**滑らかな曲線**であるため、すべての点で微分可能です。これにより、ニューラルネットワークの逆伝播（バックプロパゲーション）で誤差の傾き（勾配）をスムーズに伝えることができ、モデルが正しく学習できるようになります。

## 4. PyTorchの dim（次元）引数の意味

`torch.softmax(tensor, dim=...)` の `dim`（Dimension: 次元）は、**「どの方向（軸）に沿ってソフトマックス（合計が1.0になる計算）を適用するか」**を指定する引数です。

### ① 今回のケース（1次元ベクトルの場合）
*   `attn_scores_2` は `[6]` という形状の**1次元テンソル**（ベクトル）です。
*   軸が1つ（0番目の軸）しかないため、**`dim=0`** を指定します。
*   「この6つの要素全体を対象に合計1.0になるよう正規化する」という意味になります。

### ② 2次元行列の場合（dimの違いが最もよくわかる例）
以下のような `3行 x 4列` の行列（2次元テンソル）に対してソフトマックスを適用する場合を考えます。

```text
    [ 1.0,  2.0,  3.0,  4.0 ]  (0行目)
    [ 2.0,  0.0,  1.0,  5.0 ]  (1行目)
    [ 3.0,  1.0,  2.0,  6.0 ]  (2行目)
```

#### 縦方向 (`dim=0`): 行（Row）を潰す方向
*   `dim=0` を指定すると、**「各列（たて方向）」**の合計がそれぞれ 1.0 になるように計算されます。
*   列ごとにアテンション（重み）を計算したいときに使われます。

```text
       縦方向に計算 (dim=0)
       ↓     ↓     ↓     ↓
    [ 1.0,  2.0,  3.0,  4.0 ]
    [ 2.0,  0.0,  1.0,  5.0 ]
    [ 3.0,  1.0,  2.0,  6.0 ]
       ↓     ↓     ↓     ↓
    合計1.0 合計1.0 合計1.0 合計1.0  (各列ごとに正規化される)
```

#### 横方向 (`dim=1`): 列（Column）を潰す方向
*   `dim=1` を指定すると、**「各行（よこ方向）」**の合計がそれぞれ 1.0 になるように計算されます。
*   LLMの実装で、**「各単語が、同じ文の中のどの単語にどれくらい注目しているか（行ごとの確率分布）」**を求める際は、この `dim=-1`（最後の次元＝ヨコ方向）が多用されます。

```text
       横方向に計算 (dim=1)
    [ 1.0,  2.0,  3.0,  4.0 ]  ──> 合計 1.0 に正規化
    [ 2.0,  0.0,  1.0,  5.0 ]  ──> 合計 1.0 に正規化
    [ 3.0,  1.0,  2.0,  6.0 ]  ──> 合計 1.0 に正規化
```

## 5. 数値的不安定性（オーバーフローとアンダーフロー）と解決策

素朴なソフトマックスの実装（`softmax_naive`）をそのまま使うと、計算機が扱える数値の限界によって、計算がクラッシュして `NaN`（非数）になってしまう問題が発生します。

### ① オーバーフロー (Overflow)
*   **現象**: 数値が大きすぎて、コンピュータのメモリで表現できる限界（上限）を超えてしまうこと。
*   **原因**: ソフトマックスで使う指数関数 $e^x$ は、値が急激に巨大化します。例えば $x = 1000$ のとき、 $e^{1000}$ は無限大（`inf`）になってしまいます。
*   **結果**: 分母や分子が `inf` になり、結果として `inf / inf` のような計算が行われ、計算結果が **`NaN`（計算不可能）** になります。

### ② アンダーフロー (Underflow)
*   **現象**: 数値が小さすぎて、表現できる限界（下限）を下回り、強制的に `0.0` になってしまうこと。
*   **原因**: 入力に極端に小さなマイナス値（例: $x = -1000$）が含まれていると、 $e^{-1000}$ は限りなくゼロに近づき、コンピュータ上では `0.0` とみなされます。
*   **結果**: もしすべての入力の指数値が `0.0` になると、分母（合計値）が `0` になります。これにより **「ゼロ除算 (Zero Division)」** が発生し、やはり結果が `NaN` になります。

---

### 🛠️ 対策：安定化ソフトマックス (Stable Softmax)

PyTorchの `torch.softmax` の内部では、この問題を解決するために、数学的な「引き算のトリック」を使用しています。

#### トリックの仕組み
入力のリストの中から**「最大の数値（$x_{\text{max}}$）」を見つけ、すべての入力からそれを引き算してから** $e^x$ を計算します。

$$\text{Softmax}(x_i) = \frac{e^{x_i - x_{\text{max}}}}{\sum_{j} e^{x_j - x_{\text{max}}}}$$

*   **なぜこれで解決するのか？**:
    1.  全ての入力から $x_{\text{max}}$ を引くため、最大値だった要素の入力は必ず `0` になります。 $e^0 = 1.0$ なので、分子が `inf`（無限大）になることは絶対にありません（**オーバーフローの防止**）。
    2.  分母（合計）の中に必ず $1.0$ 以上の値（$e^0$）が含まれることになるため、分母が `0` になることもありません（**アンダーフローによるゼロ除算の防止**）。
*   数学的には、分子と分母の両方に $e^{-x_{\text{max}}}$ を掛けているだけなので、計算結果は元のソフトマックスと完全に一致します。
