# グローバルパディングとデータコレーター（動的パディング）の違い

書籍の40ページで紹介されている **「グローバルなパディング（静的パディング）」** と、今後登場する **「データコレーター（動的パディング）」** の違いと、それぞれのメリット・デメリットを整理して解説します。

---

## 1. 前提：なぜパディング（穴埋め）が必要なのか？

テキストデータは、文章ごとに長さ（単語数）がバラバラです。

* 文章A: "I love cats" (3単語)
* 文章B: "I have a cute dog and a green parrot" (8単語)

しかし、ニューラルネットワーク（GPU）で高速に並列計算を行うためには、データを綺麗な長方形（同じサイズ）の行列（テンソル）にする必要があります。そのため、短い文章の後ろに `[PAD]`（ID: 0）を付け足して長さを揃えます。この長さを揃える処理の「タイミング」に2つの方法があります。

---

## 2. グローバルパディング (静的パディング)

**「データセット全体のなかで、最も長い文章」に合わせて、すべてのデータを一律で同じ長さにパディングする方法**です。

```text
【データセット全体】
  文章A: "I love cats" (3) ───────────────> [ A, [PAD], [PAD], [PAD], [PAD], [PAD], [PAD], [PAD] ] (8)
  文章B: "I have a cute dog and a..." (8) ──> [ B ] (8)

  ※ 全データが最初から一律「長さ 8」に固定されます。
```

* **メリット**:
  * シンプルです。すべてのデータが最初から同じ形状（固定次元）になっているため、扱いやすく、テンソルへの変換も容易です。
* **デメリット (無駄が多い)**:
  * 例えば、データセットの中に「1つだけ300単語の超長文」があり、残りの1万件がすべて「平均15単語の短文」だった場合、すべての短文が無理やり300単語の長さまで無駄な `[PAD]`（0 の山）で埋められます。
  * GPUは無駄な `0` に対しても計算を行ってしまうため、**メモリを激しく浪費し、計算速度も著しく低下**します。

---

## 3. データコレーターによる動的パディング (Dynamic Padding)

データセット側はパディングせず（バラバラな長さのまま）保存しておき、**「学習時にその場で取り出したミニバッチ（例: 32件）の中だけで、最も長い文章に合わせてパディングする」** 方法です。

この、データセットからデータをミニバッチとして束ねる際に、**「そのバッチ内の最大長を調べて動的にパディングを行う仲介役」** を果たすプログラムのことを **データコレーター (Data Collator)** と呼びます。

```text
【バッチA (短い文ばかりの32件)】
  バッチ内の最大長は「12」 ──> バッチA内のデータはすべて「長さ 12」に揃えられる。

【バッチB (長めの文が混ざった32件)】
  バッチ内の最大長は「45」 ──> バッチB内のデータはすべて「長さ 45」に揃えられる。
```

* **メリット (非常に効率的)**:
  * 必要最小限の `[PAD]` トークンしか付与されないため、GPUのメモリ消費量を最小限に抑え、学習速度を大幅に高速化できます。現代のLLMのファインチューニングでは、この動的パディングが標準です。
* **デメリット**:
  * バッチごとにテンソルの形状（次元数）が毎回変わるため、手動で処理を書こうとするとコードが少し複雑になります（※Hugging Faceは `DataCollatorWithPadding` などの便利クラスを提供しているため、実務上の負担はほぼありません）。

---

## 4. 比較まとめ

| 比較項目 | グローバルパディング (静的) | データコレーターによる動的パディング |
| :--- | :--- | :--- |
| **パディング基準** | データセット全体の最大長 | 各ミニバッチ内の最大長 |
| **テンソルの形状** | 常に一定（固定） | バッチごとに変化（動的） |
| **GPUメモリ効率** | ❌ 悪い（無駄な0を多く計算する） | 🚀 非常に良い（無駄な0が最小限） |
| **学習スピード** | ❌ 遅い | 🚀 速い |
| **使いどころ** | データの事前分析や、特徴抽出 | モデルの学習・ファインチューニング |
