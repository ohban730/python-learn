# 重み共有 (Weight Tying) の仕組みとメリット

書籍126ページにおいて、パラメータ数を計算する際に**「1億6,300万個あるはずなのに、なぜ実際は1億2,400万個なのか」**という疑問が提示され、その解決策として **「重み共有 (Weight Tying)」** という概念が登場します。

このLLM（大規模言語モデル）の効率化における極めて重要なテクニックについて、数理的なアプローチと直感的な例えを交えて分かりやすく解説します。

---

## 1. 重み共有 (Weight Tying) とは？

重み共有とは、モデルの入り口（インプット）と出口（アウトプット）にある、以下の2つの巨大な重み行列を**完全に同一のメモリ実体（同じテンソル）として使い回す**設計手法です。

1.  **トークン埋め込み層 (Token Embedding / `self.tok_emb`)**
    *   **役割**: 単語ID（50,257通り）を、768次元の意味ベクトルに変換する。
    *   **重みの形状**: `torch.Size([50257, 768])`
2.  **線形出力層 (Linear Output Head / `self.out_head`)**
    *   **役割**: 768次元の意味ベクトルを、単語ID（50,257通り）の予測スコア（ロジット）に引き延ばす。
    *   **重みの形状**: `torch.Size([50257, 768])`

```text
  【入力側】  単語ID  ──( self.tok_emb )──>  768次元ベクトル
                                                  │
                                                  ▼  (トランスフォーマーブロック)
                                                  │
  【出力側】  単語ID  <──( self.out_head )──  768次元ベクトル

  ★ 入力と出力で、全く同じ重み行列 W (サイズ: 50257 × 768) を共有する！
```

---

## 2. なぜ使い回せるのか？ (数理的背景)

この2つのレイヤーは、行っていることが**「真逆（双方向）」なだけ**で、扱っている情報（「どの単語IDが、どういう意味ベクトルに対応するか」という辞書データ）は全く同一です。

*   **入力時**: $x$ というIDに対応する意味ベクトルを辞書から**「引き出す」**。
*   **出力時**: 作成された意味ベクトルが、辞書内のどの単語と一番似ているか内積を計算して**「照合する」**。

「入力用の辞書」と「出力用の辞書」を別々に用意するのではなく、**同じ1つの辞書 $W$ を共有し、出力時はその転置行列 $W^T$ を使って計算を行えばいいのではないか**、という発想からこの設計が生まれました。

---

## 3. 重み共有による3つの決定的なメリット

### ① メモリ（VRAM）の劇的な節約
語彙サイズ `50,257` × 隠れ次元数 `768` ＝ **約3,860万パラメータ** です。
これを別々に用意すると、この2つの層だけで `3860万 × 2 ＝ 約7,720万個` のメモリ領域を消費してしまいます。

重みを共有させることで、**約3,860万個分（FP32精度なら約154MB）のパラメータメモリを丸々浮かせることができます。**
これが、書籍の計算でパラメータ数が `1.63億 ──> 1.24億` へとスマートに削減されている理由です。

### ② 学習効率の大幅な向上 (対称性の獲得)
例えば、モデルが `"apple"` という単語の学習をするとき：
*   入力として `"apple"` を読み込んだとき（埋め込み層）
*   出力として `"apple"` を予測したとき（出力層）

この両方のフェーズで、**同一のパラメータが同時に更新**されます。
これにより、「入力された時の単語の意味」と「出力される時の単語の意味」がバラバラに学習されるのを防ぎ、少ないデータ量からでも非常に効率的かつ頑健に「対称的な単語の意味空間」を学習できるようになります。

### ③ 過学習 (Overfitting) の抑制
モデルが持つパラメータの「自由な無駄遣い（余計な表現力）」を抑えることで、学習データだけを丸暗記してしまう過学習を防ぎ、新しく見る未知の文章に対する予測性能（汎化性能）を向上させることができます。

## 4. 直感的な例え：1冊の「対訳対応表」

あなたが日本語と英語を翻訳して勉強しているとします。

*   **重み共有を「しない」場合**:
    *   日本語から英語に訳すための「和英辞書（入力用）」と、
    *   英語から日本語に訳すための「英和辞書（出力用）」を別々の本として用意し、それぞれに別々にメモを書き込んで勉強します。同じ単語について、両方の本に同じメモを書く必要があり非効率です。
*   **重み共有を「する」場合**:
    *   机の真ん中に**「単語の対訳対応表」が書かれた1枚の紙**を置きます。
    *   日本語から英語にするときも、英語から日本語にするときも、この同じ対応表を見ます。
    *   新しく単語の意味を覚えてその対訳表にメモを書き込めば、それは「入力」のときも「出力」のときも**自動的に両方に反映されます。**

これが、重み共有（Weight Tying）がもたらす効率化のイメージです。

## 5. PyTorchでの具体的な実装コード

PyTorchでは、モデルの初期化処理（`__init__`）の最後に、以下のように1行を記述するだけで簡単に重み共有を実現できます。

```python
class GPTModel(nn.Module):
    def __init__(self, cfg):
        super().__init__()
        self.tok_emb = nn.Embedding(cfg["vocab_size"], cfg["emb_dim"])
        self.out_head = nn.Linear(cfg["emb_dim"], cfg["vocab_size"], bias=False)
        
        # 重みの共有 (同じテンソルオブジェクトを指すように代入する)
        self.out_head.weight = self.tok_emb.weight
```

### ⚠️ 注意点
このとき、`self.out_head` のバイアス（切片）は `bias=False` に設定しておく必要があります。
埋め込み層 `nn.Embedding` はバイアスパラメータを持っていないため、重み（`weight`）だけを綺麗に共有させるためにバイアスをオフにして行列積を行うのが、GPT-2等における標準的な実装となっています。
