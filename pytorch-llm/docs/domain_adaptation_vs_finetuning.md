# ドメイン適応とファインチューニングの違いと現代のLLMにおける変遷

書籍に記載されている ULMFiT の3つのステップ（事前学習、ドメイン適応、ファインチューニング）を踏まえ、「ドメイン適応とファインチューニングの違い」や「なぜ最近のモデル（GPT-2以降）ではドメイン適応の記述があまり見られないのか」について解説します。

---

## 1. ドメイン適応 と ファインチューニング は別物？

結論から言うと、**ULMFiTの定義においては、明確に異なる2つのステップ（別物）**として扱われています。

広義にはどちらも「転移学習（モデルの調整）」に含まれますが、その目的とアプローチが異なります。

| 項目 | ② ドメイン適応 (Domain Adaptation) | ③ ファインチューニング (Fine-tuning) |
| :--- | :--- | :--- |
| **学習の目的** | モデルが扱う**「データ（領域・文脈）」**を特化させる | モデルが解く**「タスク（目的）」**を特化させる |
| **解くタスク** | **言語モデルのまま**（次の単語予測） | **別のタスクに変更**（感情分類、翻訳など） |
| **使用するデータ** | ターゲット領域の**ラベルなしテキスト**（映画レビューなど） | ターゲットタスクの**ラベル付きデータ**（正解付きデータ） |
| **目的のイメージ** | 映画業界の専門用語や独特の言い回しに「慣れる」 | 「この映画レビューはポジティブか？」に「答える」 |

* **なぜ分けるのか？**
  * Wikipediaで学習した「一般的な国語力」を持つモデル（事前学習）は、いきなり「映画レビューの感情分類」を解こうとしても、映画レビュー独特の表現（例: "Sick"が「最高」という意味で使われるなど）や専門用語を理解していません。
  * そのため、まずは「映画レビューのテキスト」をたくさん読ませて言葉遣いに慣れさせ（ドメイン適応）、その後に分類問題を解かせる（ファインチューニング）という段階を踏むことで、高い精度を達成しています。

---

## 2. なぜ GPT-2 などの解説では「ドメイン適応」の話がないのか？

以前読まれたGPT-2などの本にドメイン適応の記載がなかったのは、**モデルとデータのスケールが巨大化したことにより、わざわざ「ドメイン適応」のステップを踏む必要性が薄れたから**です。

理由は大きく3つあります。

### ① 事前学習データが最初からあらゆるドメインをカバーしている
* **ULMFiT (2018年)**: Wikipedia（学術的できれいな文章）などの限定的なデータでしか事前学習していませんでした。そのため、レビュー文やSNSなどの別ドメインに適用するには「ドメイン適応」が必須でした。
* **GPT-2以降 / 現代のLLM**: インターネット全体のコピー（Common Crawl）、本、コード、対話ログ、論文など、ありとあらゆるドメインのテキスト（数テラ〜数十テラバイト）を「事前学習（ステップ1）」の段階で丸ごと学習しています。
* つまり、**最初からあらゆる分野の専門用語や俗語に適応しているため、後から個別にドメイン適応させる必要がなくなりました**。

### ② インコンテキストラーニング (In-Context Learning) の登場
GPT-2やGPT-3は、そもそもパラメータの重みを更新するファインチューニング自体を行わなくても、プロンプトの中で「数例のデモ（Few-shot）」を見せるだけで、その場でドメインやタスクに適応して動く能力（インコンテキストラーニング）を持っています。

### ③ ファインチューニングの目的の変化
現代のLLMにおけるファインチューニングは、「特定のタスク（分類など）に特化させる」ことよりも、**「人間と対話できるように振る舞いを合わせる（指示微調整 / Instruction Tuning やアライメント）」**ことが主目的になっています。

---

## 3. 現在の転移学習は洗練されている？

**はい、非常に洗練され、手法がシンプルになりました。**

ただし、ULMFiTの「ドメイン適応」という考え方自体が消え去ったわけではありません。現代では名前を変えて、以下のような場面で使われています。

1. **継続事前学習 (Continual Pre-training)**
   * 一般的なLLMに、さらに「高度な法律」「医学」「自社の社内データ」といった極端に特化した専門知識を叩き込みたい場合、事前学習済みのモデルに対して、その専門テキストでさらに「次の単語予測」の学習を行います。これはまさにULMFiTの「ドメイン適応」そのものです。
2. **RAG (検索拡張生成)**
   * ドメインデータをモデルの重みとして学習（適応）させる代わりに、検索エンジンを使って必要なドキュメントを都度プロンプトに差し込む手法です。現代では、学習コストや情報の正確性の観点から、モデル自体をドメイン適応させるよりもRAGを使う方が主流になっています。
