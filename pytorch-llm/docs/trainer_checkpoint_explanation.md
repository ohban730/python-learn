# Trainer による「`checkpoint-500`」フォルダの作成理由と制御方法

ファインチューニング実行時に作成された `distilbert-base-uncased-finetuned-emotion/checkpoint-500` フォルダの意味と役割について解説します。

---

## 1. 結論：重複やエラーではなく「学習途中のセーブポイント（バックアップ）」です

このフォルダはモデルが重複して壊れてできたものではなく、`Trainer` が自動的に作成した **学習途中のセーブポイント（Check Point）** です。

---

## 2. なぜ「`checkpoint-500`」という名前なのか？

今回の学習条件：
* 訓練データ件数: **16,000 件**
* バッチサイズ (`per_device_train_batch_size`): **64**
* 1エポック（データ1周）のステップ数: `16,000 ÷ 64 ＝ 250 ステップ`
* 学習エポック数 (`num_train_epochs`): **2 エポック**
* **全学習ステップ数**: `250 ステップ × 2 エポック ＝ 500 ステップ`

`eval_strategy="epoch"`（エポックごとに評価）を指定しているため、エポックの区切り（今回の場合、最終ステップである **500ステップ目**）に達したタイミングで、`Trainer` がモデルの状態を保存して作成されたのが `checkpoint-500` フォルダです。

---

## 3. チェックポイントが存在する理由

1. **障害対策（Resume機能）**:
   学習が何十時間もかかる巨大モデルの場合、途中で停電やエラーで中断しても、`checkpoint-500` などのセーブデータから学習を再開（Resume）できます。
2. **ベストモデルの選択**:
   エポックごとにチェックポイントを残しておくことで、学習が進みすぎて過学習（Overfitting）が起きた際に、「一番性能が良かった時点のセーブデータ」に巻き戻すことができます。

---

## 4. チェックポイント保存の制御方法 (不要な場合の設定)

ディスク容量を節約したい場合、`TrainingArguments` で保存動作を制御できます。

### ① 中間保存を一切行わない場合
```python
training_args = TrainingArguments(
    output_dir=model_name,
    save_strategy="no",  # ★ セーブポイントを作成しない設定
    ...
)
```

### ② 直近の 1 つだけ残し、古いチェックポイントを自動削除する場合
```python
training_args = TrainingArguments(
    output_dir=model_name,
    save_total_limit=1,  # ★ 最新の1個だけを保持する
    ...
)
```
