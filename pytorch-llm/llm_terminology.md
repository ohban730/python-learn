# LLM重要用語・概念の整理：アテンション、トランスフォーマー、コンテキスト

LLM（大規模言語モデル）のコードを実装・読解する上で、混同しやすい重要な専門用語や概念の定義を整理したドキュメントです。

---

## 1. アテンション (Attention) と トランスフォーマー (Transformer) の違い

結論から言うと、**「アテンション」は計算の仕組み（部品）であり、「トランスフォーマー」はその部品を主役に据えて作られたモデル全体の構造（アーキテクチャ）**です。

```text
【自動車に例えると】
・アテンション ＝「エンジン（画期的な駆動部品）」
・トランスフォーマー ＝「そのエンジンを搭載して新設計された新型車」
```

### ① アテンション (Attention / 注意機構)
*   **定義**: 入力されたテキストの中で、**「どの単語とどの単語が強く結びついているか（どこに注意を向けるべきか）」**を動的に計算する数式の仕組み。
*   **歴史**: トランスフォーマーが登場する前（RNN/LSTMの時代）から、翻訳の精度を上げるための「追加パーツ」として存在していました。

### ② トランスフォーマー (Transformer)
*   **定義**: アテンション機構のみを組み合わせて設計された、ニューラルネットワーク全体の構造（アーキテクチャ）。
*   **特徴**: 2017年の論文 *“Attention Is All You Need”* で発表されました。それまで主流だった **RNN（再帰型ニューラルネット）を一切使わず、アテンションだけで構成した** ことで、計算の並列化が可能になり、学習速度と精度が劇的に向上しました。現在の GPT や Claude などのベースになっています。

## 2. 二つの「コンテキスト (Context)」

「コンテキスト」という言葉は、話の文脈によって物理的な「サイズ」を指す場合と、データとしての「意味ベクトル」を指す場合があります。

### ① コンテキストサイズ (Context Size / 最大長)
*   **意味**: モデルが一度に処理（入力・考慮）できる**テキスト（トークン）の物理的な範囲・視野の広さ**。
*   **コード上の変数**: `max_length` や `context_length`。
*   **例え**: カメラの「画角」や、読書するときの「一目で視野に入る範囲」。

### ② コンテキストベクトル (Context Vector / 文脈ベクトル)
この言葉には、モデルのアーキテクチャによって2つの異なる使われ方（定義）があります。

*   **定義A：【Self-Attention における定義】（本書での定義）**
    *   **意味**: 各入力単語に対して計算される、**「他のすべての単語の情報をミックスして強化された埋め込みベクトル」**。
    *   *例*: `"journey"` という単体の埋め込みベクトル $x^{(2)}$ に対し、文内の他の単語との関連度（アテンションの重み）を掛け合わせて合成したベクトル $z^{(2)}$。
    *   *特徴*: 単語の「辞書的な意味」に「周辺の文脈」が追加された、**強化されたベクトル**であり、入力されたトークンの数だけ存在します。
*   **定義B：【従来のRNN/Seq2Seq における定義】**
    *   **意味**: 入力テキスト全体（文全体）を読み込み、その**意味をギュッと「1つのベクトル」に圧縮したデータ**（文全体のサマリー）。
    *   *使われ方*: エンコーダーからデコーダーへ文脈情報を引き渡す際に使われていました。
*   **コンテキスト消失（ボトルネック）の解決とアテンション**:
    *   従来のRNNモデルでは、どれほど長い入力文であっても、最後の「たった1つのコンテキストベクトル」にすべての情報を無理やり圧縮していました。そのため、長い文章では文頭の方の情報（コンテキスト）が途中で消えてしまう「コンテキストの消失」が発生していました。
    *   アテンション機構は、デコーダーが出力する際、入力テキストの**すべての単語（過去の隠れ状態）に対して、その都度どれが重要かを計算して直接アクセス（選択的アクセス）**できるようにしました。これにより、長距離の複雑な文章が入力されても、コンテキストが失われることなく正確に処理できるようになりました。

## 3. エンコーダー (Encoder) と デコーダー (Decoder)

テキストを処理する役割の違いを指します。

```text
[入力: 日本語] ──> 【エンコーダー】 ──(コンテキスト)──> 【デコーダー】 ──> [出力: 英語]
                     (意味を理解・圧縮)                      (文脈から翻訳を生成)
```

### ① エンコーダー (Encoder / 符号化器)
*   **役割**: 入力されたテキスト（例: 翻訳元の日本語）を読み込み、その意味を分析して**ベクトルに変換する（理解する）**役割。

### ② デコーダー (Decoder / 復号化器)
*   **役割**: エンコーダーが作成したベクトル（意味）を受け取り、それを元に**新しいテキスト（例: 翻訳先の英語）を生成する**役割。

> 💡 **GPTシリーズは「デコーダーのみ」のモデル**
> 翻訳モデルは「エンコーダー＋デコーダー」の構成ですが、GPTのようなテキスト生成AIは、入力されたプロンプトの「続き」を生成するタスクに特化しているため、**デコーダー部分のみ**を積み重ねた構造をしています。

## 4. 隠れ状態 (Hidden State)

*   **意味**: モデルの内部（中間層）で計算の途中段階に生成される、単語の**一時的なベクトル表現（中間データ）**のことです。
*   **「埋め込みベクトル」との違い（ステージの違い）**:
    *   **入力埋め込み (Input Embedding)**: モデルに入る前の、文脈を考慮していない「辞書的な単語の意味（最初のベクトル）」。
    *   **隠れ状態 (Hidden State)**: モデルの各層（レイヤー）を通過し、アテンションなどを経て**「周囲の文脈情報を吸収した単語の意味（途中のベクトル）」**。
    *   *例*: `"Apple is a tech company."` の `"Apple"` と、`"I ate an apple."` の `"Apple"` は、最初の**入力埋め込みは同じ**ですが、レイヤーを通過した後の**隠れ状態は異なる数値**（文脈が反映されたベクトル）になります。
*   アテンション機構は、この「過去のすべての隠れ状態（＝過去の単語が持っていた文脈付きの意味ベクトル）」に直接アクセスして、どの単語に注目すべきかを計算します。

## 5. シーケンス (Sequence)

*   **意味**: 直訳すると「順序」「並び」。機械学習やLLMにおいては**「順番に並んだデータ（トークンの列 / 文章）」**のことを指します。
*   **テキストとシーケンス**: テキストは、単語がバラバラに存在するのではなく、必ず「並び順（時系列）」を持っています。この「順序を持つデータ」を表現する言葉としてシーケンスが使われます。
    *   *例*: `"I love learning AI."` は、5つのトークンが並んだ「長さ5のシーケンス（文章）」です。
*   **関連用語**:
    *   **シーケンス長 (Sequence Length)**: シーケンスに含まれるトークンの総数（コード上では `seq_len` や `x.shape[1]` と表現されます）。
    *   **自己アテンション (Self-Attention)**: 1つのシーケンス「内」（例: `"I love learning AI."` の文の中だけ）で単語同士の関連性を計算する仕組み。

## 6. 数学の次元 vs プログラミング（テンソル）の次元

「3次元ベクトル」や「2次元のテンソル」という言葉を聞いた時、数学の視点とプログラミングの視点で「次元」という言葉の意味が異なるため、混同が生じやすくなります。

### ① 数学的な次元 (要素の数 / 幾何学的な空間の次元)
*   **定義**: ベクトルの中にある**「要素（数値）の数」**。
*   *例*: `[0.55, 0.87, 0.66]`
    *   3つの数値（x, y, z座標など）で構成されているため、数学的（幾何学的）には**「3次元のベクトル」**と呼びます。

### ② プログラミングにおける次元 (テンソルの軸の数 / 配列の深さ)
*   **定義**: 配列の**「括弧のネストの深さ（軸の数 / 階数 / Rank）」**。PyTorchでは `ndim` や `dim()` で取得される値です。
*   *例A (1次元テンソル / ベクトル)*: `[0.55, 0.87, 0.66]`
    *   括弧が1重なので、PyTorch（プログラミング）では**「1次元テンソル」**と呼びます（Shapeは `[3]`）。
*   *例B (2次元テンソル / 行列)*: `[[0.55, 0.87, 0.66], [0.1, 0.2, 0.3]]`
    *   括弧が2重なので、PyTorchでは**「2次元テンソル」**と呼びます（Shapeは `[2, 3]`）。

> 💡 **なぜ `torch.dot` でエラーが起きるのか？**
> `torch.dot(a, b)` は、「**プログラミング的な次元数 (ndim) が両方とも 1** のテンソル（＝1次元の配列）同士」のドット積しか計算できないというルールがあります。
> `query` は `[0.55, 0.87, 0.66]`（1重の括弧、Shape: `[3]`）なので **1次元テンソル** として動作しますが、`inputs`（2重の括弧、Shape: `[6, 3]`）は **2次元テンソル** であるため、ルール違反となりエラーが発生します。
